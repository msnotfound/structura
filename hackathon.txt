Core Stack

LangGraph · LangChain · MCP Server · kubectl · Prometheus · Slack

"When production breaks at 3am, engineers spend 40 minutes grepping logs.
K8sWhisperer does it in 90 seconds — autonomously."

The Problem
Modern Kubernetes clusters are complex, dynamic, and unforgiving. When a production pod crashes, an on-call
engineer receives a PagerDuty alert, SSHes into the cluster, reads logs, cross-references metrics, hypothesises
root causes, and executes a fix — a process taking 30–60 minutes of high-stress manual work, even for
experienced engineers.
The tools to automate every individual step already exist: kubectl reads cluster state, Prometheus tracks
metrics, LangGraph orchestrates multi-step agent logic, and MCP servers expose tools to LLMs. The challenge
is integrating all of these into a coherent, safe, and explainable autonomous system — one that acts confidently
when safe, escalates when uncertain, and never destroys what it is trying to protect.

Goal
Build K8sWhisperer — an autonomous Kubernetes incident response agent that continuously monitors
a cluster, detects anomalies, diagnoses root causes, plans remediation actions, and executes fixes —
while routing risky actions through a human-in-the-loop approval flow. Every decision must be logged in
plain English.

01

Observe — Cluster
scan

kubectl MCP polls all namespaces every 30 seconds — events, pod

03

AND action not in DESTRUCTIVE_ACTIONS → auto-execute. Otherwise
→ HITL Slack approval.

06

state to confirm resolution. Sets result field in ClusterState.

message. Appends LogEntry to persistent audit_log JSON. Loops back to

Hidden Traps — What Will Break Naive Implementations

HIGH

Race Conditions

Two anomalies fire simultaneously on the same pod — your agent must handle
concurrent state updates without corrupting ClusterState.

False Positives

A pod restarts during a rolling update (expected) vs a CrashLoop (bad). LLM must
distinguish planned from unplanned restarts.

RBAC Footgun

If your MCP tool has cluster-admin, a hallucinated action can delete namespaces.
Scoping to pod-level delete only is non-trivial to implement correctly.

Verify Loop

After restarting a pod, it may take 60+ seconds to become Running. Verify step must poll
with backoff, not just check once.

Slack Webhook

HITL approval can take minutes. LangGraph must pause the graph cleanly and resume

Latency

on callback — not spin-wait or timeout prematurely.

Log Noise

kubectl logs returns megabytes. Passing the full log to the LLM hits context limits.
Chunking and summarisation strategy required.

• Constraint Solving — Enforce geometric constraints: wall closure, alignment, consistency. Infer missing
dimensions from context.
• Real-Time Material Pricing — Integrate a live construction material price API instead of the static database.
• Multi-Storey Support — Extend the pipeline to handle stacked floor plans with inter-floor structural
dependencies.
• Cost Breakdown Report — Generate a line-item material cost estimate for the full structure, exportable as
PDF.

Hidden Traps — What Will Break Naive Implementations
Non-90° Layouts

Even "clean" floor plans may have slight angular deviations. Snapping coordinates to
an orthogonal grid without tolerance checks produces impossible geometries.

Junction Detection

T-junctions and L-corners look similar in line detection output. Misclassifying them
creates open room boundaries that break polygon extraction.

3D Handoff Quality

If wall detection produces slightly offset coordinates, the 3D model will have gaps or
floating walls. One bad coordinate cascades into an unusable model.

Load-Bearing

A rule like "outer walls are load-bearing" is necessary but not sufficient. Central

Classification

structural spines, columns, and beam positions require additional heuristics.

Material Tradeoff Formula

A naive formula (cost + strength) weights both equally. Judges will probe whether
teams thought about why weight ratios differ between structural and non-structural
elements.

Explainability Depth

An LLM that says "Red Brick is recommended because it is good" will score poorly.
Explanations must cite specific element properties and span measurements.

Evaluation Rules

• Teams must demo a working system live — recorded demos are accepted only as backup if technical
failure occurs during presentation.
• Web3 bonus marks are awarded only if integration is functional and meaningful — storing a random hash
with no connection to the pipeline output → 0 marks.
• Use of any LLM API (OpenAI, Anthropic, Mistral, open-source models) is permitted and not penalised.
• Teams using the structural PS fallback manual coordinate clause must disclose this during demo —
dishonest disclosure results in 0 marks for the parsing criterion.
• For K8sWhisperer: judges will inject a live anomaly during the demo. The team must let the agent respond
autonomously — no manual kubectl commands during the demo window.
• Plagiarism or direct use of existing open-source Kubernetes operator code without significant modification
→ disqualification.
• All teams submit: source code (GitHub link), README with setup instructions, and architecture diagram
before the judging session begins.

The real challenge is not the technology. Any team can stitch together APIs. The
challenge is building a reasoning system that knows when to act, when to ask, and when
to stop — and an analysis system that turns raw geometry into genuine insight. That is
what separates a script from an agent, and a parser from an engineer.
